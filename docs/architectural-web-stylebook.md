## 1. Purpose

This stylebook defines the architectural language for applications, dashboards, workspaces, operating environments, and future systems inside the ProfoundProjects and Sentient OS ecosystem.

It does not replace the Design Genome foundation. It extends it by governing:

- spatial behavior
- visual construction
- environmental language
- structural hierarchy
- surface logic
- light behavior
- circulation models
- architectural consistency
- operating atmosphere

## 2. Core Principle

Sentient OS does not build websites.

Sentient OS builds operating environments.

Every interface is architecture:

- interface = architecture
- workspace = room
- dashboard = floor
- navigation = circulation
- panel = structural surface
- launcher = entrance lobby
- data view = observation deck
- feed = signal corridor
- modal = temporary chamber
- command bar = operational control surface

Every visual decision must improve at least one of the following:

- orientation
- hierarchy
- understanding
- focus
- movement
- decision making
- system confidence

Decoration without purpose is forbidden.

## 4. Non-Negotiable Rules

All decisions follow this order:

1. Foundation first
2. Architecture second
3. Personality third
4. Project DNA fourth
5. Application fifth
6. Feature last

Hard rules:

- no feature may invent a new visual language
- no application may bypass Foundation
- no project may create isolated UI behavior unless documented as an exception
- no personality may break accessibility, spacing, typography hierarchy, or component contracts
- no visual effect may exist only because it looks good
- no screen may feel like a loose collection of cards
- every interface must feel like a constructed environment

## 5. Design Philosophy

The user should never feel like they are looking at random widgets.

The user should feel like they are entering a digital space with:

- structure
- purpose
- flow
- orientation
- hierarchy
- atmosphere
- operational meaning

Every interface should answer:

- Where am I?
- What environment am I inside?
- What is the current state?
- What is important?
- What can I do next?
- Where can I go?
- What needs my attention?
- What is stable?
- What is changing?

The goal is not sameness.

The goal is recognizable operational consistency across the ecosystem.

## 6. Architectural Construction Layers

Every screen is built through six layers.

### Layer 1: Space

A screen is a spatial environment, not a page.

Approved spatial identities:

- Command Center
- Control Tower
- Operations Deck
- Executive Gallery
- Glass Atrium
- Research Lab
- Knowledge Library
- Signal Terminal
- Blueprint Studio
- Creative Workshop
- Production Floor
- Horizon Workspace
- Launch Lobby
- System Observatory

Every page must declare a spatial identity before UI generation begins.

### Layer 2: Structure

Structure defines load-bearing hierarchy and orientation.

Approved structural elements:

- App Shell
- Header
- Navigation Rail
- Sidebar
- Command Bar
- Workspace
- Page Header
- Content Grid
- Panel Group
- Intelligence Rail
- Context Drawer
- Detail Surface
- Status Strip
- Metric Row
- Timeline Floor
- Launcher Grid

Rules:

- navigation must remain predictable
- the workspace must always be visually dominant
- the command area must clearly communicate available actions
- the intelligence rail must support context, not distract from it
- panel groups must follow a clear hierarchy
- a screen may have many elements, but only one primary focus

### Layer 3: Materials

Materials communicate function, weight, and atmosphere.

Approved material directions:

- Graphite
- Dark Metal
- Soft Glass
- Frosted Glass
- Ceramic White
- Blueprint Surface
- Carbon Layer
- Matte Surface
- Ambient Surface
- Executive Glass
- Terminal Plane
- Paper Light
- Studio Canvas

Rules:

- dark surfaces communicate control, monitoring, and technical focus
- light surfaces communicate planning, overview, lifestyle, and accessibility
- glass surfaces communicate premium overview and layered intelligence
- matte surfaces communicate focus and reduced distraction
- blueprint surfaces communicate design, production, and construction logic
- terminal surfaces communicate precision and engineering control

Materials may change atmosphere.

Materials may not break component contracts.

### Layer 4: Light

Light is hierarchy, not decoration.

Approved light systems:

- Command Lighting
- Executive Daylight
- Horizon Light
- Terminal Glow
- Ambient Illumination
- Signal Highlight
- Blueprint Glow
- Atrium Reflection
- Focused Spotlight
- Night Observatory

Rules:

- use light to guide attention
- use glow only for state, focus, intelligence, or important activity
- do not add glow to every component
- do not create decorative lighting noise
- high-priority information may receive stronger light contrast
- secondary information should remain visually calm
- danger, warning, success, and active states must be immediately legible

### Layer 5: Circulation

Movement must feel intentional.

Approved circulation paths:

- Launcher to Workspace
- Navigation Rail to Page
- Command Bar to Action
- Dashboard to Detail Panel
- Feed to Item Detail
- Metric to Analysis
- Project to Timeline
- Agent to Execution State
- Knowledge Node to Document
- Alert to Resolution Path

Rules:

- users must always know how to return
- primary navigation must remain stable
- deep views must preserve context
- details should open in panels, drawers, routes, or focused workspaces based on complexity
- important actions should never be hidden behind unclear icon-only controls
- old paths must redirect safely when navigation evolves

### Layer 6: Atmosphere

Atmosphere must support the user's objective.

Approved atmosphere modes:

- Operational
- Executive
- Creative
- Technical
- Focused
- Research
- Lifestyle
- Performance
- Premium
- Calm
- Urgent
- Analytical

Guidance:

- finance should feel calm, premium, and decisive
- runtime systems should feel operational, structured, and alive
- developer workspaces should feel precise, efficient, and low-noise
- creative systems should feel modular, open, and exploratory
- planning systems should feel calm, human, and manageable
- agent systems should feel intelligent, observable, and controllable

## 7. Architectural Object Model

Components are architectural objects, not decorative blocks.

Object mappings:

- cards = rooms
- panels = walls
- dashboards = floors
- feeds = corridors
- navigation = circulation
- launchers = lobbies
- command bars = control surfaces
- metric cards = instruments
- data tables = ledgers
- timelines = roads
- modals = temporary chambers
- drawers = side rooms
- tabs = internal zones
- forms = intake desks
- empty states = unoccupied rooms
- knowledge maps = libraries
- agent consoles = control booths

Component rules:

- every component must have one clear responsibility
- every component must communicate its role through structure, hierarchy, and state
- no component may carry multiple competing primary actions

## 9. Synced Theme System

Before any UI generation begins, every application must declare:

- `design_personality`
- `theme_mode`
- `density_mode`
- `motion_mode`
- `chapter_accent`
- `architectural_family`
- `spatial_identity`
- `application_mode`

Theme sync rules:

- all themes inherit Foundation tokens
- all themes support Day, Night, and Auto
- all themes support Compact, Standard, Focus, and Executive density
- all themes support Minimal, Standard, and Dynamic motion
- all themes must preserve component contracts
- all themes must preserve accessibility requirements
- all themes must map to approved architectural families

A synced theme may change:

- atmosphere
- surface color
- accent behavior
- depth intensity
- motion intensity
- density expression
- environmental mood

A synced theme may not change:

- spacing scale
- typography hierarchy
- base component anatomy
- keyboard accessibility
- responsive structure
- status semantics
- navigation predictability

Foundation controls:

- AppShell

Any proposed design that violates Foundation must be refactored before implementation.

## 14. Motion Rules

Motion must clarify system behavior.

Approved motion types:

- fade
- slide
- subtle scale
- elevation shift
- panel reveal
- status transition
- route transition
- drawer movement

Motion modes:

- Minimal
- Standard
- Dynamic

Motion is allowed when it improves:

- orientation
- cause and effect
- state change
- focus transition
- progress
- system feedback

Motion is forbidden when it creates:

- noise
- confusion
- delay
- visual fatigue
- decorative distraction

## 15. Accessibility Rules

Accessibility is part of architecture.

Every theme must support:

- WCAG AA contrast minimum
- visible focus states
- keyboard navigation
- readable typography
- responsive layouts
- semantic structure
- state clarity
- reduced motion support

If atmospheric styling breaks accessibility, the atmosphere must change.

## 16. Design Evaluation Checklist

Before approval, ask:

- does it improve orientation?
- does it improve hierarchy?
- does it improve circulation?
- does it improve understanding?
- does it improve focus?
- does it strengthen the ecosystem?
- does it fit the declared architectural family?
- does it respect the selected personality?
- does it preserve Foundation?
- does it support synced themes?
- does it remain accessible?

If the answer is no, remove it, simplify it, refactor it, or isolate it as a documented exception.

## 18. Implementation Rule For Developers

Before creating a new component, check:

- can this be built with an existing Foundation component?
- can this be extended from an approved component?
- can this become a reusable ecosystem pattern?
- does this belong in Foundation, Personality, Project DNA, Application, or Feature?

Decision order:

1. Reuse first
2. Extend second
3. Create third
4. Document always

## 19. Long-Term Objective

Build a design language that is as recognizable as major platform systems while remaining specific to Sentient OS:

- shared foundation
- shared architectural language
- shared personality framework

- shared navigation philosophy

Foundation first.
