## 1. DESIGN TOKENS

### Spacing Scale (`--space-*`)

- [ ] Use `--space-1` (4 px) for tight icon gaps and compact inline spacing
- [ ] Use `--space-2` (8 px) for small control padding and close related UI groupings
- [ ] Use `--space-3` (12 px) for compact panel padding and metric label spacing
- [ ] Use `--space-4` (16 px) as the default component padding and standard content rhythm
- [ ] Use `--space-5` (24 px) for card spacing, section gaps, and comfortable reading surfaces
- [ ] Use `--space-6` (32 px) for major section separations and layout breathing room
- [ ] Use `--space-7` (48 px) for page-level padding and large vertical rhythms
- [ ] Use `--space-8` (64 px) for wide layout margins and hero spacings
- [ ] Use `--space-9` (80 px) for full-page hero vertical rhythms
- [ ] **No arbitrary spacing values** — all layout spacing must use Foundation spacing tokens

### Typography (`--text-*`, `--font-*`, `--weight-*`, `--line-*`)

- [ ] Use `--font-primary` ("Geist Variable", system-ui) for headings and primary interface text
- [ ] Use `--font-secondary` ("Inter Variable", system-ui) for secondary content and fallback reading
- [ ] Use `--font-mono` ("JetBrains Mono", monospace) for logs, code, metrics, and terminal content
- [ ] `--text-h1` (32 px) is **reserved for page title only**
- [ ] `--text-h2` (24 px) for major section headers
- [ ] `--text-h3` (20 px) for sub-section headers
- [ ] `--text-body` (16 px) as the default body reading size
- [ ] `--text-small` (14 px) for labels and captions
- [ ] `--text-caption` (12 px) for the smallest UI text
- [ ] **Decorative typography is not allowed** — readability wins

### Elevation (`--elevation-*`)

- [ ] `--elevation-0`: no shadow — flat/muted surfaces
- [ ] `--elevation-1`: default card and panel shadow
- [ ] `--elevation-2`: elevated or selected surfaces
- [ ] `--elevation-3`: overlapping drawers and sidebar overlays
- [ ] `--elevation-4`: highest elevation for critical modals and command overlays

### Radius (`--radius-*`)

- [ ] `--radius-xs` (4 px) — tight interactive elements (buttons, toggles)
- [ ] `--radius-sm` (8 px) — small cards and inputs
- [ ] `--radius-md` (12 px) — standard cards
- [ ] `--radius-lg` (16 px) — panels and sections
- [ ] `--radius-xl` / `--radius-2xl` — large surface containers
- [ ] `--radius-full` (999 px) — pill badges and circular elements

### Z-Index (`--z-*`)

- [ ] `--z-base` (0) — base workspace surfaces
- [ ] `--z-header` (100) — persistent app header
- [ ] `--z-navigation` (200) — navigation rail and sidebar
- [ ] `--z-drawer` (300) — slide-over drawers
- [ ] `--z-command-bar` (400) — pinned command surfaces
- [ ] `--z-modal` (500) — modal chambers
- [ ] `--z-toast` (600) — transient notifications
- [ ] `--z-tooltip` (700) — tooltips and ephemeral overlays
- [ ] **No arbitrary z-index values in application UI**

### Motion (`--duration-*`, `--ease-*`)

- [ ] `--duration-fast` (120 ms) — micro-interactions, state changes
- [ ] `--duration-normal` (220 ms) — panel reveals, transitions
- [ ] `--duration-slow` (360 ms) — route transitions and deliberate animations
- [ ] Use `--ease-standard`, `--ease-enter`, `--ease-exit` — no custom cubic-bezier values
- [ ] Motion must **clarify system behavior**, never decorate

### Theme Declaration (required before UI generation)

- [ ] Declare `design_personality`
- [ ] Declare `theme_mode` (Day / Night / Auto)
- [ ] Declare `density_mode` (Compact / Standard / Focus / Executive)
- [ ] Declare `motion_mode` (Minimal / Standard / Dynamic)
- [ ] Declare `chapter_accent`
- [ ] Declare `architectural_family`
- [ ] Declare `spatial_identity`
- [ ] Declare `application_mode`

## 2. UI GUARDRAILS

### Accessibility (WCAG AA minimum)

- [ ] Minimum **WCAG AA contrast** on all text and interactive elements
- [ ] Every interactive element must expose a **visible focus state** (`outline: var(--focus-ring-width) solid currentColor`)
- [ ] Full **keyboard navigation** support — no mouse-only interactions
- [ ] Semantic HTML — use `<header>`, `<nav>`, `<main>`, `<aside>`, appropriate ARIA roles
- [ ] Include a **skip-to-main-content** link for keyboard users
- [ ] Support **`prefers-reduced-motion`** — all motion is conditional
- [ ] Status states (danger, warning, success, active) must be **immediately legible**
- [ ] If atmospheric styling breaks accessibility, **the atmosphere must change** — not the accessibility rule

### Responsive Breakpoints (`--bp-*`)

- [ ] `--bp-xs` (480 px) — mobile layout
- [ ] `--bp-sm` (640 px) — large mobile
- [ ] `--bp-md` (768 px) — tablet; sidebar collapses to overlay at this breakpoint
- [ ] `--bp-lg` (1024 px) — desktop; full side-by-side layout
- [ ] `--bp-xl` (1280 px) — wide desktop
- [ ] `--bp-2xl` (1440 px) — operational dashboard width
- [ ] Mobile layout: sidebar must collapse or become a slide-over overlay
- [ ] Desktop layout: sidebar is sticky and side-by-side with the content canvas

### Synced Theme Rules (what themes MAY and MAY NOT change)

- [ ] Themes **may** change: atmosphere, surface color, accent behavior, depth, motion intensity, density, environmental mood
- [ ] Themes **may not** change: spacing scale, typography hierarchy, base component anatomy, keyboard accessibility, responsive structure, status semantics, navigation predictability

### Component Behavior Rules

- [ ] Every component must have **one clear responsibility**
- [ ] No component may carry **multiple competing primary actions**
- [ ] Navigation must remain **predictable** across state changes
- [ ] Primary navigation must remain **stable** — do not hide or move it unexpectedly
- [ ] Details should open in panels, drawers, routes, or focused workspaces — **based on complexity**
- [ ] Important actions must **never be hidden behind unclear icon-only controls**
- [ ] A screen may have many elements but only **one primary focus**

### Foundation Inheritance Rules

- [ ] Every project **must** inherit the Foundation layer — no bypassing it
- [ ] No application may bypass Foundation
- [ ] No feature may invent a new visual language
- [ ] Any design that violates Foundation must be **refactored before implementation**

## 3. PAGE DESIGN SKILLS

### Six-Layer Construction (required before screen generation)

Every screen is built through six declared layers:

- [ ] **Layer 1 — Space**: Declare a spatial identity (e.g., Command Center, Glass Atrium, Signal Terminal, Horizon Workspace)
- [ ] **Layer 2 — Structure**: Declare App Shell, Header, Navigation Rail, Sidebar, Command Bar, Workspace, Content Grid, optional Intelligence Rail, Context Drawer, Status Strip
- [ ] **Layer 3 — Materials**: Declare surface material (Graphite, Soft Glass, Frosted Glass, Ceramic White, Blueprint Surface, etc.)
- [ ] **Layer 4 — Light**: Declare light system and use light to guide attention — never decoratively
- [ ] **Layer 5 — Circulation**: Map entry, navigation, deep-view, and return paths
- [ ] **Layer 6 — Atmosphere**: Declare atmosphere mode (Operational, Executive, Creative, Technical, Focused, etc.)

### Layout Structure Rules

- [ ] Every interface must include an **App Shell** with a persistent `Header`, `Navigation`, and `Workspace`
- [ ] The workspace must always be **visually dominant**
- [ ] The command area must clearly communicate **available actions**
- [ ] The intelligence rail must **support context** — not distract from it
- [ ] Panel groups must follow a **clear hierarchy**
- [ ] Navigation must remain **sticky** and predictable — users must always know how to return
- [ ] Deep views must **preserve context** — use panels, drawers, or focused routes

### Architectural Families (declare one per screen)

- [ ] **Command Tower**: dense, structured, mission-control, instrument-driven, high-contrast — for operational control and runtime management
- [ ] **Glass Atrium**: open, premium, atmospheric, spacious — for executive overview and platform entry
- [ ] **Blueprint Studio**: modular, technical, grid-based, construction-oriented — for creation and production
- [ ] **Signal Terminal**: precise, efficient, low-noise, terminal-inspired — for engineering and agent control
- [ ] **Executive Gallery**: elegant, minimal, calm, decision-focused — for finance, strategy, reporting
- [ ] **Horizon Workspace**: calm, balanced, human, light-capable — for planning and personal productivity

### Dashboard Grid / Workspace Canvas Rules

- [ ] Use **Content Grid** (`pf-workspace-grid`) as the main widget injection surface
- [ ] Support density modes: Compact (operations), Standard (general), Focus (deep work), Executive (strategy)
- [ ] Widgets injected by AI skills or dynamic systems must conform to the **same component contracts**
- [ ] No screen may feel like a "loose collection of cards" — every surface must feel like a **constructed environment**
- [ ] Use metric cards, data tables, timelines, feed corridors, and panel groups according to their **object model role**

### Sidebar / Navigation Rail Rules

- [ ] The sidebar is a **structural circulation element** — not a decorative panel
- [ ] On desktop (≥ 768 px): sidebar is sticky and side-by-side
- [ ] On mobile (< 768 px): sidebar collapses to an overlay triggered by a toggle button
- [ ] The toggle button must expose `aria-expanded` and `aria-controls` attributes
- [ ] Sidebar collapse state must be **visually communicated** to users

### Codex / AI Output Requirements

Every Copilot or AI-generated component must return:

- [ ] Declared architectural family
- [ ] Declared spatial identity
- [ ] Declared personality
- [ ] Declared theme mode, density mode, motion mode
- [ ] Foundation compliance notes
- [ ] Component reuse notes
- [ ] Accessibility notes
- [ ] Files changed
- [ ] Design governance summary
